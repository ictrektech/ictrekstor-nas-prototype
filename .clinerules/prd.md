# PRD 文档维护规范

## 目的

PRD（Product Requirement Document）是项目各页面功能行为的权威描述。本规范确保 PRD 与页面代码保持同步，使 PRD 持续具备参考价值。

## 核心原则

- **PRD 描述功能，不描述视觉**：PRD 关注页面有哪些功能、如何交互、数据流向什么，不关注颜色、字体、间距、动效等视觉表现。视觉规范独立维护于 `docs/visual-design/`。
- **PRD 描述预期，不描述当前状态**：PRD 只描述页面**应该具备**的功能行为，不描述当前实现状态（如"已实现"、"施工中"等）。实现进展由其他文档或项目管理工具跟踪。
- **PRD 描述功能，不描述布局**：PRD 不描述页面具体布局（如"左侧目录树 + 右侧文件面板"），布局属于视觉/交互设计范畴，由视觉规范或设计稿决定。
- **PRD 是事实来源（Single Source of Truth）**：当 PRD 与代码不一致时，以 PRD 为基准修正代码，或以代码为基准修正 PRD，二者必须对齐。

## 触发 PRD 更新的场景

### 1. 页面功能变更

当对页面内容进行**功能级别**的变更时，必须同步更新对应的 PRD 文件：

| 变更类型 | 示例 | PRD 更新内容 |
|---------|------|-------------|
| 新增功能 | 我的文件页面新增"标签管理"功能 | 在功能规格章节添加对应描述 |
| 功能移除 | 移除公共文件的"创建子文件夹"功能 | 删除或标注已移除的功能描述 |
| 功能修改 | 共享设置从弹窗改为抽屉交互 | 更新功能规格和预期交互章节 |
| 数据模型变更 | Mock 数据结构变化 | 更新数据需求章节 |

**以下变更不需要更新 PRD**：
- 纯视觉调整（颜色、字体、间距、圆角、阴影、动效等）
- 页面布局调整（左右结构、上下结构、卡片排列方式等）
- 代码重构但功能行为不变
- 变量/函数命名调整但逻辑不变
- 修复不影响功能行为的 bug

### 2. 页面路由变更

当页面路由发生以下任何变更时，必须同步调整 PRD 文件及索引：

| 变更类型 | 示例 | PRD 更新内容 |
|---------|------|-------------|
| **页面重命名** | `/file/shared` → `/file/my-shares` | 修改对应 PRD 的页面概要、路由字段；更新 `docs/prd/prd.md` 导航表 |
| **页面移动** | `/storage/disks` → `/system/disks` | 移动 PRD 文件到对应目录；更新 `docs/prd/prd.md` 导航表 |
| **页面删除** | 删除不再需要的页面 | 删除对应 PRD 文件或在文件中标注已废弃；更新 `docs/prd/prd.md` |
| **页面合并** | 两个页面合并为一个 | 合并 PRD 内容并删除冗余文件；更新 `docs/prd/prd.md` |
| **新增页面** | 新增功能模块页面 | 在对应目录下新建 PRD 文件；更新 `docs/prd/prd.md` 导航表 |

**必须同步更新的文件清单**：
1. 受影响的 PRD 文件本身
2. `docs/prd/prd.md` — 总索引中的导航表格和目录结构
3. 其他引用该页面的 PRD 文件中的"关联页面"章节

## PRD 文件格式规范

### 文件位置

PRD 文件按模块组织在 `docs/prd/` 下，目录结构与路由/页面结构保持一致：

```
docs/prd/
├── prd.md              # 总索引（功能需求导航）
├── home/
│   └── index.md
├── file/
│   ├── my-files.md
│   └── ...
├── storage/
│   ├── disk-detail.md
│   └── ...
├── system/
│   ├── device.md
│   └── ...
└── ai/
    └── chat.md
```

### 必含章节

每篇 PRD 统一包含以下章节（按顺序）：

1. **页面概要** — 路由、菜单标题/图标、页面权限、页面类型
2. **需求概述** — 总体功能、目标用户、核心价值
3. **功能规格** — 以结构化、层次化方式详细列出每个功能，包含：
   - **功能描述**：功能做什么
   - **预期行为**：功能如何响应用户操作
   - **约束条件**：前置条件、限制、边界情况
   - **可用场景**：何时使用此功能
   - **涉及接口**：描述该功能需要后端提供的数据能力。每个已在 `interface.md` 中定义的接口名称需添加指向该文档对应锚点的 Markdown 链接（子目录 PRD 使用 `../interface.md#锚点`，根目录使用 `interface.md#锚点`），未定义的接口保持纯文本。不涉及具体 URL 和 HTTP 方法
4. **预期交互** — 典型用户交互流程（按场景组织）
5. **数据需求（Mock）** — 页面所需数据结构，使用前端 Mock 实现
6. **关联页面** — 与该页面有跳转或数据关联的其他页面

### 功能规格组织方式

功能规格按**功能模块 → 子功能**的层次结构组织，模块和子功能均不使用编号：

```
### 功能模块: 模块名称

#### 功能: 子功能名称
- **功能描述**: ...
- **预期行为**: ...
- **约束条件**: ...
- **可用场景**: ...
- **涉及接口**: ...

#### 功能: 子功能名称
...
```

### 预期交互组织方式

预期交互按典型用户场景组织，场景不使用编号：

```
### 场景: 场景名称
1. 步骤1
2. 步骤2
...
```

### 关于 API 与 Mock 数据

- PRD 中提到的接口仅描述功能需要后端提供的**数据能力**，使用文字描述（如"获取用户目录树结构"、"创建存储空间"），不涉及具体 URL、HTTP 方法等实现细节。
- **当前项目为纯前端原型**，没有后端 API 服务。所有数据必须**完全在前端 Mock**，禁止在原型代码中直接调用真实后端 API。
- Mock 数据应硬编码在页面内或独立的 Mock 文件中，模拟真实的数据结构和交互响应。

### 涉及接口名称一致性规则

PRD 中「涉及接口」的接口显示名称必须与 `interface.md` 中对应接口的标题名称**严格一致**。

- 若接口已在 `interface.md` 中定义，必须使用 Markdown 链接格式：`[接口标题名称](../interface.md#锚点)`，其中方括号内的文本必须与 `interface.md` 中该接口的 `###` 标题完全相同。
- 若 `interface.md` 中不存在对应接口，保持纯文本，但应评估是否需要在 `interface.md` 中补充定义。
- **禁止**使用自定义描述性文字作为链接文本（如 `[获取分享目录下的文件列表]` 应改为 `[获取文件列表]`）。
- **禁止**链接指向不存在的锚点。添加新接口链接前，务必先在 `interface.md` 中确认对应标题存在。

### 统一文件操作 API 约定

不同资源类型（如存储空间、公共文件夹、他人共享文件夹、外接设备等）在获取其**资源信息**（目录树、资源元数据）时需要独立的 API，但一旦确定了资源对应的本地路径，后续的**文件操作**应使用统一的 API，不应为每种资源类型单独定义：

| 操作类型 | 统一 API | 说明 |
|---------|---------|------|
| 获取文件列表/目录树 | [获取文件列表](../interface.md#获取文件列表) | 通过 `path` + `mode` 参数获取文件列表（list 模式）或目录树（tree 模式） |
| 创建文件夹 | [创建文件夹](../interface.md#创建文件夹) | 通过 `path` 参数定位目标目录 |
| 重命名 | [重命名文件/文件夹](../interface.md#重命名文件文件夹) | 通过 `id` 定位目标 |
| 上传 | [上传文件](../interface.md#上传文件) | 通过 `path` 参数定位目标目录 |
| 下载 | [下载文件](../interface.md#下载文件) | 通过 `id` 定位目标文件 |
| 移动/复制 | [移动/复制文件](../interface.md#移动复制文件) | 通过 `ids` + `targetPath` 定位 |
| 删除 | [删除文件](../interface.md#删除文件) | 通过 `ids` 定位目标 |
| 搜索 | [搜索文件](../interface.md#搜索文件) | 通过 `keyword` + `scope` 定位范围 |

**需保持独立的 API（资源发现/定位）**：

这些 API 用于确定资源对应的文件系统路径，是调用统一文件操作 API 的前置步骤：

| 资源类型 | 资源发现 API |
|---------|-------------|
| 我的文件（存储空间） | [获取用户存储空间列表](../interface.md#获取用户存储空间列表) |
| 公共文件夹 | [获取公共文件夹列表](../interface.md#获取公共文件夹列表) |
| 他人分享 | [获取他人给当前用户的分享列表](../interface.md#获取他人给当前用户的分享列表) |
| 外接设备 | [获取外接设备列表](../interface.md#获取外接设备列表) / [获取外接设备详情](../interface.md#获取外接设备详情) |
| 本地目录（外接设备管理） | [获取本地目录树](../interface.md#获取本地目录树) |

**禁止的做法**：
- 不要为每种资源类型定义独立的"创建文件夹"、"重命名"、"上传"、"下载"等文件操作 API。
- PRD 中涉及接口应优先链接到统一的文件操作 API，若当前 interface.md 中存在重复定义，应合并为统一 API。

## Agent 执行规范

当 Agent 执行涉及页面功能或路由的变更时：

1. **变更前**：检查是否存在对应 PRD 文件，了解当前功能描述
2. **变更后**：判断变更是否触发上述 PRD 更新场景，如触发则同步更新 PRD
3. **索引同步**：任何路由/页面级别的变更完成后，检查并更新 `docs/prd/prd.md`
4. **关联同步**：检查其他 PRD 的"关联页面"章节是否需要同步更新

## 索引文件维护

`docs/prd/prd.md` 是 PRD 总索引，包含：
- 项目概要介绍
- 核心概念详解（存储池、存储空间、硬盘、文件管理模型、共享等）
- 各模块 PRD 导航表格（路由 ↔ PRD 文件）
- PRD 目录结构
- 维护说明

**维护要求**：
- 新增/删除/移动 PRD 文件后，必须同步更新索引中的导航表格和目录结构
- 核心概念发生变更时，需同步更新概念说明章节
